这篇文字来源主要是 Wiki，详情见 参考[1]，以及其他网络资料，我努力做到把 Restful 解释清楚，尽可能使语言通俗易懂，逻辑顺序友好，方便阅读理解。

历史

Roy Fielding 在 O'Reilly Open Source Convention 发表演讲 2008.

Roy Fielding 在他2000年的博士论文“架构风格和基于网络的软件架构设计”中定义了 REST。他在现有的1996年 HTTP 1.0设计的基础上，与1996-1999年的 HTTP 1.1并行开发了 REST 架构风格。

Fielding 的论文解释了1994年开始被称为“ HTTP 对象模型”的 REST 原则，并用于设计 HTTP 1.1和统一资源标识符(URI)标准。这个术语的目的是让人想象一个设计良好的 Web 应用程序是如何运作的: 它是一个 Web 资源网络(一个虚拟状态机) ，用户通过选择资源标识符(比如 http://www.example.com/articles/21/ )和资源操作(比如 GET 或 POST)在应用程序中前进，从而将下一个资源的表示(下一个应用程序状态)转移到最终用户使用。

“网络资源”最初是在万维网上定义为通过网址确定的文件或文件。 然而，今天它们有了一个更加通用和抽象的定义，包含了每一个事物、实体或行为，这些事物、实体或行为可以在网络上以任何方式被识别、命名、处理、处理或执行。 在 RESTful Web 服务中，对资源的 URI 发出的请求将引发响应，其有效负载格式为 HTML、 XML、 JSON 或其他格式。 响应可以确认对资源状态做了一些更改，响应可以提供到其他相关资源的超文本链接。 当使用 HTTP 时(这是最常见的) ，可用的操作(HTTP 方法)是 GET、 HEAD、 POST、 PUT、 PATCH、 DELETE、 CONNECT、 OPTIONS 和 TRACE。

定义

REST是“呈现状态转移（REpresentational State Transfer）”的缩写。或许可以这样来定义它：一种API的架构风格，在客户端和服务端之间通过呈现状态的转移来驱动应用状态的演进。

通过使用无状态协议和标准操作，RESTful 系统的目标是快速的性能、可靠性，以及通过重用组件来增长的能力，这些组件可以在不影响整个系统的情况下进行管理和更新，即使系统在运行。

体系结构的限制

六个指导性约束定义了一个 RESTful 系统。 这些约束限制了服务器处理和响应客户机请求的方式，因此，通过在这些约束下运行，系统获得了理想的非功能属性，如性能、可伸缩性、简单性、可修改性、可视性、可移植性和可靠性。 如果一个系统违反了任何所需的约束，它就不能被认为是 RESTful。

CS模式

客户机-服务器约束背后的原理是关注点分离。 将用户界面关注点与数据存储关注点分离，可以提高用户界面跨多个平台的可移植性。 它还通过简化服务器组件来提高可伸缩性。 也许对 Web 最重要的是，这种分离允许组件独立进化，从而支持多个组织领域的 internet 规模需求。

无状态

客户端上下文在多个请求之间是绝不会保存在服务器上的。每个请求必须包含必要的信息。无状态的服务器通过快速释放资源和简化实现提高了可伸缩性。可靠性使得从局部失败中恢复变得容易。很明显，监控系统不必通过考虑单个请求来判断请求的性质。

无状态服务器的一个缺点是降低了网络性能，因为所有需要的数据必须在每次请求中发送。

客户机-服务器通信由于请求之间没有客户机上下文存储在服务器上而受到限制。 来自任何客户机的每个请求都包含为请求提供服务所需的所有信息，会话状态在客户机中保存。 会话状态可以由服务器传输到另一个服务(如数据库) ，以便在一段时间内保持持久状态并允许身份验证。 当客户机准备好转换到新状态时，它开始发送请求。

可缓存

REST应用程序是web系统，因此客户端和中间层可以缓存响应。响应必须被定义为可缓存或不可缓存的，以防客户端重复使用旧数据导致降低可靠性。如果缓存中的陈旧数据与已生成的请求的数据显著不同，则由服务器处理请求。缓存可以消除一些客户端和服务器之间的交互，这就提升了可伸缩性、效率和通过减少平均延迟达到的用户可感知的性能。

分层的系统

客户机通常无法判断它是直接连接到终端服务器，还是连接到中间层。 如果在客户机和服务器之间放置一个代理或负载均衡器，它不会影响它们之间的通信，也不需要更新客户机或服务器代码。 中间服务器可以通过启用负载平衡和提供共享缓存来提高系统的可伸缩性。 此外，安全性可以作为 web 服务的顶层添加，然后将业务逻辑与安全逻辑清楚地分开。将安全性作为一个单独的层来添加，可以强制执行安全策略。 最后，它还意味着服务器可以调用多个其他服务器来生成对客户机的响应。

按需编码(可选)

服务器可以通过传输可执行代码来临时扩展或定制客户端的功能: 例如，已编译的组件(如 javaapplet)或客户端脚本(如 JavaScript)。

统一的接口

统一的接口约束是任何 RESTful 系统设计的基础。 它简化和解耦了体系结构，使每个部分能够独立进化。 这一统一界面的四个限制因素是:

  1. 请求中的资源标识

  2. 通过表示进行资源操作

  3. 自我描述性信息

  4. 超媒体作为应用程序状态的引擎

最后两个其实我也没弄明白怎么回事，照搬 Wiki 原文，以后有机会再琢磨吧。

总结：架构特性

Rest 体系结构样式的约束影响以下体系结构属性:

  • 组件交互性能，这是影响用户感知性能和网络效率的主导因素

  • 可扩展性

    允许支持大量的组件和组件之间的交互。 Roy Fielding 将 REST 对可伸缩性的影响描述如下：

    Rest 的客户机-服务器关注点分离简化了组件实现，降低了连接器语义的复杂性，提高了性能调优的有效性，并增加了纯服务器组件的可扩展性。 分层的系统约束允许在通信的各个点引入中介体ー代理、网关和防火墙，而不改变组件之间的接口，从而允许它们协助通信转换或通过大规模共享缓存提高性能。 Rest 通过约束消息的自描述性使中间处理成为可能: 请求之间的交互是无状态的，标准方法和媒体类型用于表示语义和交换信息，响应明确表示可缓存性。

  • 统一界面的简单性

  • 组件的可修改性，以满足不断变化的需求(即使在应用程序运行时)

  • 服务代理之间通信的可见性

  • 通过将程序代码与数据一起移动来实现组件的可移植性

  • 可靠性，在组件、连接器或数据中出现故障时，系统层面的抗故障能力
